# 프론트엔드 과제 평가 가이드

## 공통 평가 항목

| 영역 | 평가 포인트 |
|---|---|
| 코드 품질 | 컴포넌트 분리 기준이 합리적인가, 관심사 분리가 되어 있는가 |
| 커스텀 훅 | 반복 로직(타이머, 폼 검증 등)을 훅으로 추출했는가 |
| 타입 안전성 | TypeScript 사용 시 any 남용 없이 적절한 타입 정의를 했는가 |
| 렌더링 최적화 | 불필요한 리렌더링을 인지하고 memo/useMemo/useCallback을 적재적소에 사용했는가 |
| UX 디테일 | 로딩, 에러, 비활성 상태 등 엣지 케이스를 빠뜨리지 않았는가 |

## 3. 턴제 배틀 게임

### 폼 평가

| 항목 | 구체적으로 보는 것 | 왜 중요한가 |
|---|---|---|
| 스텝 폼 설계 | 3단계 폼의 진행/후퇴가 자연스러운가, 각 스텝의 상태가 보존되는가 | 다단계 폼 아키텍처 |
| 연쇄 초기화 | 클래스 변경 시 스킬 선택이 함께 리셋되는가 | 의존 상태 간 동기화 |
| 조건부 스킬 풀 | 클래스별로 다른 스킬 목록이 정확히 렌더링되는가 | 데이터 기반 동적 폼 |
| 다중 선택 + 상한 | 스킬 3개 제한이 정확히 동작하고, 초과 시 적절한 피드백이 있는가 | 선택 제한 로직 |
| 커스텀 스킬 생성 폼 | 타입 선택에 따라 입력 필드가 조건부로 전환되는가 (공격→배율, 회복→회복량, 버프/디버프→스탯+수치+턴) | 동적 폼의 핵심 — 타입 기반 조건부 렌더링 |
| 커스텀 스킬 유효성 검증 | 이름, MP, 배율, 수치, 턴 등 각 필드의 범위 제한이 정확한가 | 복합 폼 검증 |
| 커스텀 스킬 CRUD | 추가한 스킬이 목록에 반영되는가, 삭제 시 선택 상태도 연동 해제되는가 | 동적 리스트 상태 관리 |

### 상태 관리 평가

| 항목 | 구체적으로 보는 것 | 왜 중요한가 |
|---|---|---|
| 전투 상태 구조 | 플레이어/적의 HP, MP, 버프, 방어 등을 어떤 구조로 관리하는가 | 복잡한 도메인 상태 설계 |
| 턴 시스템 | 턴 시작(독/버프 틱) → 행동 → 턴 종료 순서가 정확한가 | 순서 의존적 상태 업데이트 |
| 단일 효과 원칙 | 각 스킬이 정확히 하나의 효과만 실행하는가 (공격만, 디버프만 등 혼합 없음) | 스킬 시스템 설계의 명확성 |
| 버프/디버프 턴 관리 | 잔여 턴 감소, 만료 제거, 동일 버프 갱신(중첩 불가)이 정확한가 | 시간 기반 상태 관리의 핵심 |
| 유효 스탯 계산 | 기본 스탯 + 버프 합산이 실시간으로 UI에 반영되는가 | 파생 상태 계산 |
| 조건부 행동 | MP 부족 시 스킬 비활성화, 방어 시 피해 감소가 정확한가 | 상태 기반 UI 분기 |
| AI 의사결정 | 난이도별로 다른 패턴을 보이는가, HP 비율에 따라 전략이 바뀌는가 | 상태 기반 로직 분기 |
| 전투 로그 | 모든 행동이 빠짐없이 기록되고, 타입별로 구분 가능한가 | 이벤트 히스토리 관리 |
