# 프론트엔드 과제 평가 가이드

## 공통 평가 항목

| 영역 | 평가 포인트 |
|---|---|
| 코드 품질 | 컴포넌트 분리 기준이 합리적인가, 관심사 분리가 되어 있는가 |
| 커스텀 훅 | 반복 로직(타이머, 폼 검증 등)을 훅으로 추출했는가 |
| 타입 안전성 | TypeScript 사용 시 any 남용 없이 적절한 타입 정의를 했는가 |
| 렌더링 최적화 | 불필요한 리렌더링을 인지하고 memo/useMemo/useCallback을 적재적소에 사용했는가 |
| UX 디테일 | 로딩, 에러, 비활성 상태 등 엣지 케이스를 빠뜨리지 않았는가 |

## 지뢰찾기

### 폼 평가

| 항목 | 구체적으로 보는 것 | 왜 중요한가 |
|---|---|---|
| 조건부 렌더링 | 프리셋 ↔ 커스텀 전환 시 입력 필드가 정확히 나타나고 사라지는가 | 폼 상태와 UI 동기화 기본기 |
| 상호 의존 검증 | 격자 크기 변경 시 최대 지뢰 수가 동적으로 재계산되는가 | 파생 상태 관리 능력 |
| 에러 처리 | 범위 초과 입력에 대한 에러 메시지가 정확한 위치에 즉시 표시되는가 | 사용자 피드백 설계 |
| 상태 초기화 | 프리셋 변경 시 커스텀 값이 올바르게 리셋되는가 | 폼 상태 라이프사이클 이해도 |

### 상태 관리 평가

| 항목 | 구체적으로 보는 것 | 왜 중요한가 |
|---|---|---|
| 2D 배열 설계 | 보드 상태를 어떤 자료구조로 설계했는가 (불변성 유지 방식) | 복잡한 데이터 구조의 상태 관리 |
| 재귀 로직 | 빈 셀 클릭 시 연쇄 열기가 정확하게 동작하는가 (BFS/DFS) | 알고리즘 + 상태 업데이트 결합 |
| 첫 클릭 안전 보장 | 첫 클릭 셀과 주변 8칸에 지뢰가 없도록 보장하는가 | 게임 공정성 + 조건부 보드 생성 |
| 승리 조건 판정 | 지뢰가 아닌 모든 셀이 열렸는지를 어떻게 판별하는가 (깃발 꽂기 불필요) | 종료 조건의 정확한 구현 |
| 패배 원인 구분 | 지뢰 클릭 vs 타임아웃을 구분하여 다른 피드백을 제공하는가 | 사용자 피드백 설계 |
| 이벤트 분기 | 좌클릭/우클릭이 충돌 없이 각각 정확히 동작하는가 | 이벤트 핸들링 숙련도 |
| 게임 상태 머신 | idle → ready → playing → won/lost 전환이 명확하고 가드가 존재하는가 | 상태 머신 설계 |
| 타이머 정리 | 컴포넌트 언마운트 시 타이머가 정확히 정리되는가 | useEffect 클린업 이해도 |

### 렌더링 최적화 평가

| 항목 | 구체적으로 보는 것 | 왜 중요한가 |
|---|---|---|
| 셀 단위 메모이제이션 | React.memo + 커스텀 비교 함수로 변경된 셀만 리렌더링되는가 | 최대 480셀(Expert) 렌더링 성능 |
| 보드 평탄화 캐싱 | board.flat() 결과를 useMemo로 캐싱하는가 | 매 렌더마다 O(n) 연산 방지 |
| 동적 CSS Grid | 보드 크기에 따라 gridTemplateColumns/Rows가 동적으로 설정되는가 | 다양한 보드 사이즈 대응 |
